1. System Overview
1.1 High-Level Architecture
CardVault is a multi-tenant SaaS platform with four distinct tiers:
Client Tier: React Native Android APK (OCR on-device via Google ML Kit) + React/TS Admin Dashboard
API Tier: NestJS REST API behind reverse proxy with JWT middleware and rate limiting
Service Tier: BullMQ job workers (OCR, dedup, image processing, sync, relationship matching)
Data Tier: PostgreSQL 15 (Supabase managed, RLS-enforced) + Supabase Storage (S3-compatible)

1.3 Main Technical Workflows
Capture: Mode select → OCR (on-device ML Kit) → review/qualify → relationship check → contact write + encounter append → session counter increment
Offline Sync: Local SQLite queue → background sync on reconnect → idempotency-keyed ingestion → session counter reconciliation
Relationship Match: Post-ingest job → fuzzy match on email/phone/name+company → push history card to mobile → user decision (add encounter / new / view)
Export: Manager triggers per-session CSV export → BullMQ export-generation job → signed download URL returned
Auth: Login → Supabase Auth → JWT (RS256, 1hr TTL) + refresh token (30d, rotating) → org_id + role claims on every request

3.4 Offline Sync Architecture
State
Trigger
Action
OFFLINE
Network lost
All writes → local SQLite (session_id, capture_mode, qualifier, encounter_type)
CONNECTING
Network restored
Background sync service activates; session counters begin reconciliation
SYNCING
Service running
FIFO queue drain; image upload; OCR submit; session counter sync
CONFLICT
Server 409
Server-wins; local version archived in sync_queue
COMPLETE
Queue empty
Sync timestamp updated; relationship checks run; UI refresh
ERROR
Repeated failure
Move to dead-letter queue; surface in Sync Status screen
Idempotency: every offline capture carries a client-generated client_idempotency_key (UUID v4). Server deduplicates on ingest, returning HTTP 200 if already processed.
Session counter reconciliation: server_count + unsynced_offline_captures = reconciled total. Handled by session-counter-sync BullMQ queue on each sync cycle.

4.4 Multi-Tenant Considerations
Every table includes organization_id as a non-nullable FK. RLS policies enforce: SELECT/INSERT/UPDATE/DELETE restricted to rows where organization_id = auth.jwt()->>'organization_id'.
audit_events is append-only — no UPDATE or DELETE RLS granted.
event_sessions and contact_encounters have organization_id with RLS; cross-tenant access is blocked at DB layer before API layer.
Storage bucket paths are org-namespaced: {org_id}/sessions/{session_id}/{contact_id}/.
At 100M+ contacts: partition contacts by organization_id hash. contact_encounters partition by created_at year at 50M+ rows.
scan_count hot-path mitigation: increment buffered in Redis, flushed to PG in 5-second batches (avoids row-lock contention during high-volume exhibitor scans).

6. Core Functional Flows
6.1 Visitor Flow
Tap Visitor → check for active Visitor session in sessionStore
No active session: Join existing (org-shared list) or Create new (name + event_type + date + optional GPS location)
Camera opens with session name in header; session_id scoped to capture
Post-OCR review: editable fields + optional note → save → contact.event_session_id = session.id
Relationship match job fires async; history card displayed if match found
Session view: scrollable list of contacts in session, filterable by date/note
6.2 Exhibitor Flow
Tap Exhibitor → create booth session (name + event_type + date range + team assignment)
Multiple team members share one session_id; scan counts aggregate server-side
Post-OCR: Lead Qualifier screen (HOT/WARM/COLD large tap targets) + 160-char note
Auto-advance after 3s if no note entered (speed mode); camera reopens immediately
Live session summary: total, HOT, WARM, COLD, unqualified counts (Redis-buffered, 5s flush)
Manager: export session CSV filtered by qualifier tier
6.3 Quick Capture Flow
Tap Quick Capture → Encounter type selector (7 tiles; default = last-used)
Camera opens; post-OCR: encounter type (pre-filled, editable) + 200-char note + optional follow-up date
Follow-up date → local device notification (no third-party push without opt-in)
Contact saved with encounter_type; no session_id (null); org-wide visible
6.4 Relationship History Flow
Post-OCR ingest → relationship-matching BullMQ job → fuzzy match on normalized email, phone, name+company (Levenshtein ≤0.85 threshold, scoped to organization_id)
Match found: push to client with prior session names, dates, captured_by, qualifier
User choice: (a) Add encounter → new contact_encounters row appended; (b) Create new → independent contact created; (c) View existing → navigate to contact detail
Contact detail screen: full encounter history timeline ordered by created_at DESC
6.5 Offline Sync Flow
Capture offline → write to pending_captures (SQLite) with all v3.0 fields
On reconnect: sync service drains queue FIFO → POST /api/v1/ocr/submit with client_idempotency_key
Server: check idempotency_key in cache → 200 (already processed) or 201 (new)
Session counter: session-counter-sync job reconciles local count vs server count
Relationship matching deferred until sync; batched relationship checks post-sync
7. Security Architecture
7.1 JWT Auth
Algorithm: RS256 (asymmetric); private key in secrets manager
Claims: sub (user_id), org (organization_id), role, iat, exp (1h), jti
Refresh tokens: 30-day expiry, rotated on every use; previous token invalidated
Token revocation: jti blocklist in Redis with TTL = remaining token expiry
Auth rate limit: 5 failed attempts per 15 min per IP → 429 with Retry-After

7.3 Tenant Isolation
Supabase RLS on every table: SELECT/INSERT/UPDATE/DELETE WHERE organization_id = auth.jwt()->>'organization_id'
TenantContextMiddleware extracts org claim and sets request-scoped tenant context before any service call
Cross-tenant API requests rejected at TenantGuard with 403 before query execution
Storage bucket paths org-namespaced; bucket-level policies block cross-tenant object access
Session data (event_sessions, session_members, contact_encounters) fully org-isolated via RLS
7.4 Encryption & Transport
Data at rest: AES-256 via Supabase managed encryption (PostgreSQL + Storage)
Transport: TLS 1.3 enforced; TLS 1.2 minimum fallback; HSTS 1-year max-age
Certificate pinning in React Native for API domain
Signed URLs for image access: 1-hour TTL, HTTPS only
Lead notes and encounter notes subject to same AES-256 encryption at rest as all PII
7.5 Audit Logging
All auth events, contact mutations, session creates/closes, exports, and role changes written to audit_events
audit_events is append-only; no UPDATE or DELETE RLS granted to any role
Export events include: actor identity, timestamp, filter criteria, qualifier filter, record count
Relationship history decisions logged: match confidence, contact IDs, user choice, timestamp

1. Architecture Overview
1.1 Backend Architecture
CardVault is a four-tier multi-tenant SaaS platform. All backend state lives in PostgreSQL 15 (Supabase managed). The API tier is stateless and horizontally scalable.
Tier
Technology
Responsibility
Client
React Native 0.74+ / React 18 + TS
Mobile capture, on-device OCR, offline queue, admin dashboard
API
NestJS / Node.js 20 LTS
Business logic, RBAC, auth middleware, rate limiting, REST endpoints
Workers
BullMQ + Redis (Upstash)
Async OCR, dedup, image processing, relationship matching, export, audit
Data
PostgreSQL 15 (Supabase) + Supabase Storage
Multi-tenant RLS-enforced data + S3-compatible object storage
1.2 PostgreSQL + Supabase Setup
Managed PostgreSQL 15 with PgBouncer in transaction mode.
Row Level Security (RLS) enforced on every table — no exceptions.
Supabase Auth provides RS256 JWT with org_id and role claims.
Supabase Storage (S3-compatible, AES-256 at rest) for card images.
Separate Supabase project per environment (dev / staging / prod).
Read replica used for analytics queries to offload primary.
1.3 Redis / BullMQ
Redis (Upstash Serverless) — ephemeral state only. Nothing durable stored solely in Redis.
Uses: rate limiting counters, JWT jti blocklist, scan_count buffer, BullMQ queue backing.
scan_count hot-path: Redis INCR per scan → flushed to PG every 5 seconds in batch (prevents row-lock at 1+ scan/sec exhibitor throughput).
8 named BullMQ queues with independently scaled worker pools.
1.4 Multi-Tenant Strategy
Tenant key: organization_id UUID on every table.
RLS policy on every table: WHERE organization_id = auth.jwt()->>'organization_id'.
TenantContextMiddleware extracts org claim before any service call.
TenantGuard rejects cross-tenant requests at API layer (403) before DB query.
Storage paths namespaced: org/{org_id}/sessions/{session_id}/{contact_id}/.
1.5 Offline-First Sync Strategy
Mobile SQLite stores full capture payload in pending_captures table.
Every capture carries client_idempotency_key (UUID v4).
On reconnect: sync service drains FIFO via POST /ocr/submit with idempotency key.
Server-wins on 409 conflict; local version archived in sync_queue with status=dead_letter.
Session counter reconciliation: session-counter-sync BullMQ job runs post-sync cycle.
2. Database Schema
All tables include organization_id (FK → organizations), created_at, updated_at. Soft-delete uses deleted_at TIMESTAMPTZ (nullable). RLS enforces tenant isolation on every table.

6. Multi-Tenant & RLS Design
6.1 Tenant Isolation Strategy
Every table carries organization_id UUID NOT NULL (with FK to organizations).
organization_id is extracted from JWT claim: auth.jwt()->>'organization_id'.
TenantContextMiddleware (NestJS) sets request-scoped tenant before service calls.
TenantGuard rejects mismatched org in URL vs JWT claim with 403 before any DB query.
Storage paths are org-namespaced; bucket policies enforce cross-tenant isolation.

9.3 Fuzzy Match Optimization
pg_trgm extension with GIN index on (full_name || company) enables fast similarity queries.
Match query scoped to organization_id first to minimize comparator set.
Email/phone exact match checked first (O(1) via GIN array index); fuzzy match only if no exact hit.
relationship_matches table records every match result for threshold tuning analytics.
Match job triggered async post-OCR; does not block contact creation response.

14. Security Considerations
14.1 JWT Validation
Algorithm: RS256 (asymmetric). Private key in secrets manager (never in code/env).
Claims validated: sub, org, role, exp, jti. Invalid or missing claims → 401.
jti blocklist in Redis with TTL = remaining token expiry (checked before DB access).
Refresh tokens: 30-day expiry, rotated on every use; previous invalidated.
Certificate pinning in React Native APK for API domain.

15. Quick Reference — Schema Change Checklist
Apply to every new table or column addition in a multi-tenant migration.
Add organization_id UUID NOT NULL REFERENCES organizations(id) to every new table.
Enable RLS: ALTER TABLE <table> ENABLE ROW LEVEL SECURITY.
Create SELECT, INSERT, UPDATE policies scoped to auth.jwt()->>'org'.
Determine if table needs soft-delete (deleted_at TIMESTAMPTZ) or hard-delete.
Add created_at NOT NULL DEFAULT NOW(); add updated_at if mutable.
Determine trigger for updated_at maintenance (trigger function or application-managed).
Add composite index (organization_id, <primary_filter_column>) as minimum.
If full-text searchable: add TSVECTOR GENERATED column + GIN index.
If dedup-critical arrays (emails, phones): add GIN index on array column.
If high-volume insert table: evaluate Redis buffer → batch flush pattern.
If append-only (audit, encounters): omit UPDATE/DELETE RLS policies.
Add BullMQ job if mutation requires async side effects (OCR, dedup, audit).
Register entity_type in audit_event_type_enum and audit-writes queue payload.

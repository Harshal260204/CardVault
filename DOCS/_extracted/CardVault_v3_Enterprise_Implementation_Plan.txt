1. Executive Engineering Summary
1.1 System Goals
CardVault v3.0 is a production-grade, multi-tenant enterprise SaaS platform for business card intelligence. The system digitizes, centralizes, and operationalizes business card data at enterprise scale across three context-aware capture modes: Visitor, Exhibitor, and Quick Capture. The platform must operate reliably in degraded-network environments (trade show floors, airports, conference venues) while providing real-time analytics and relationship intelligence to enterprise managers.
The core engineering objectives are:
Zero contact data loss: offline captures must survive indefinitely with guaranteed sync-on-reconnect
Multi-tenant isolation: PostgreSQL RLS enforces organizational boundaries at the data layer—no exceptions
Async pipeline reliability: OCR, deduplication, and relationship matching must handle failure gracefully without blocking user workflows
Sub-second capture experience: camera-to-saved < 8 seconds including on-device OCR
Audit completeness: every mutation, export, and auth event immutably recorded

2.3 Request Lifecycle
Every authenticated API request traverses the following middleware chain in order:
TLS termination at load balancer / Cloudflare
Rate limiter middleware: Redis sliding window check per user/endpoint
JwtAuthGuard: RS256 token verification, jti blocklist check in Redis
TenantContextMiddleware: Extract organization_id from JWT claims, attach to request context
TenantGuard: Validate URL-scoped org parameters match JWT org claim (403 on mismatch)
RolesGuard: Verify role claim satisfies endpoint RBAC requirement
DTO validation: class-validator pipe, field-level constraint enforcement
Service layer: business logic execution, repository calls
PostgreSQL RLS: Final data-layer enforcement (WHERE organization_id = auth.jwt()->>'org')
Response interceptor: Envelope wrapping { data, meta, error }, correlation-id injection
Exception filter: Normalize errors to standard format, Sentry capture on 5xx
2.4 OCR Lifecycle
The OCR pipeline is the critical path for the core product value proposition. It spans on-device processing through async server-side enrichment:
Quality gate (on-device): Camera screen validates resolution ≥ 800×600, blur score (Laplacian variance), tilt angle. Failed gate prompts retake.
Auto-shutter (on-device): When on-device ML Kit confidence > 90% for 800ms, camera auto-captures with haptic + flash feedback.
ML Kit processing (on-device): Google ML Kit Text Recognition V2 extracts text fields with per-field confidence scores (0.0–1.0).
Signed URL request: POST /images/upload-url returns 15-minute signed S3 PUT URL; card_images row created with status=pending.
Direct S3 upload: Client uploads original JPEG to Supabase Storage (path: org/{org_id}/sessions/{sess_id}/{contact_id}/original.jpg).
Image confirm: POST /images/:id/confirm updates status=confirmed, enqueues image-processing BullMQ job for thumbnail generation (400×300 JPEG q70).
OCR submit: POST /ocr/submit with full payload (mode context, session_id, qualifier, encounter_type, OCR fields, confidence scores, client_idempotency_key). Idempotency key checked in Redis cache (TTL 24h) before DB lookup.
Contact creation: contacts and contact_encounters rows written synchronously within OCR submit handler.
Async enrichment: deduplication job enqueued → relationship-matching job enqueued post-dedup (if no exact match found).
Response: 201 (new contact) or 200 (idempotency replay) returned immediately; relationship match result pushed via push notification when available.
2.5 Sync Lifecycle (Offline-First)
The offline sync system guarantees zero data loss for captures taken without connectivity:
Offline detection: React Native NetInfo detects connectivity loss; syncStore state machine transitions ONLINE → OFFLINE.
Local capture: Full payload (image URI, OCR fields, mode context, qualifier, client_idempotency_key) written to SQLite pending_captures table. UI shows 'Saved locally' with amber sync badge.
Reconnect detection: NetInfo fires CONNECTING → SYNCING; background sync service activates.
Image upload: Each pending capture uploads image first (retry-able; same signed URL flow). Image must confirm before OCR submit.
FIFO drain: pending_captures drained in created_at order. Each POST /ocr/submit includes client_idempotency_key.
Server response handling: 201 = new; 200 = already processed (mark complete, no duplicate); 409 = conflict (server-wins, local version archived to sync_queue with status=conflict).
Dead-letter: After 3 retries, sync_queue.status = dead_letter. Surfaced in Sync Status screen as 'failed · Retry'.
Counter reconciliation: session-counter-sync BullMQ job runs post-sync, computes server_count + unsynced_count = reconciled total.
Completion: UI transitions to ONLINE, relationship checks run for all synced offline captures.

Turborepo is recommended over Nx for CardVault v3.0. 4.3 Key Middleware and Guards
TenantContextMiddleware must run before any guard or service. It extracts organization_id from the decoded JWT payload and attaches it to the request context object. This is the single point of org claim extraction—all subsequent service calls read from request.tenantContext.organizationId, never re-parsing the JWT.
TenantGuard validates that any :orgId, :sessionId, or :contactId URL parameter belongs to the calling user's organization. It does this by looking up the entity in the repository with an organization_id filter before passing control to the service layer. This prevents IDOR (Insecure Direct Object Reference) attacks where an attacker substitutes IDs from other organizations.
RolesGuard reads the @Roles() decorator metadata from the route handler and compares against the JWT role claim. Role hierarchy: super_admin > manager > employee. Insufficient role returns 403 with an audit log entry.
5. PostgreSQL + Supabase Design
5.1 Why PostgreSQL + Supabase
PostgreSQL 15 is the correct choice for CardVault because: (1) RLS is a first-class PostgreSQL feature, enabling database-level multi-tenant isolation without application-layer filtering that could be bypassed; (2) GIN indexes support both tsvector full-text search and array-contains queries (emails[], phones[]) in a single engine; (3) pg_trgm extension enables fuzzy string matching (Levenshtein/trigram) required by the relationship intelligence engine; (4) Generated columns (search_vector TSVECTOR GENERATED ALWAYS AS STORED) eliminate application-side index maintenance.
Supabase is appropriate because it provides: managed PostgreSQL 15 with PgBouncer in transaction mode, Supabase Auth with RS256 JWT including org/role claims, S3-compatible object storage with AES-256 at rest, and a production-ready read replica. The managed service reduces operational burden for a startup-phase product while maintaining full PostgreSQL compatibility.
5.2 RLS Policy Design
Every table has RLS enabled with at minimum a SELECT policy. The standard pattern:
-- Enable RLS on every table
ALTER TABLE contacts ENABLE ROW LEVEL SECURITY;
-- SELECT: users see only their org
CREATE POLICY contacts_select ON contacts
FOR SELECT USING (organization_id = (auth.jwt()->>'organization_id')::UUID);
-- INSERT: users can insert into their org
CREATE POLICY contacts_insert ON contacts
FOR INSERT WITH CHECK (organization_id = (auth.jwt()->>'organization_id')::UUID);
-- UPDATE: employees can only update own captures; managers can update org-wide
CREATE POLICY contacts_update ON contacts
FOR UPDATE USING (
organization_id = (auth.jwt()->>'organization_id')::UUID
AND (created_by = (auth.jwt()->>'sub')::UUID
OR (auth.jwt()->>'role') IN ('manager','super_admin'))
);
-- audit_events: append-only, no UPDATE/DELETE policy
CREATE POLICY audit_insert ON audit_events
FOR INSERT WITH CHECK (true);  -- service role only writes

5.4 Migration Strategy
Migrations use Supabase CLI numbered migration files (supabase/migrations/YYYYMMDDHHMMSS_description.sql). Rules:
Migrations are always additive in production. No DROP COLUMN, DROP TABLE, or destructive ALTER in a migration without a preceding deprecation phase.
New nullable columns are safe to add without downtime. New NOT NULL columns require a default or backfill migration.
RLS policy changes run in a transaction. Policy drop + recreate must be atomic.
New indexes: CREATE INDEX CONCURRENTLY to avoid table locks on active production tables.
The schema change checklist (from schema doc section 15) is enforced as a PR template gate.
5.5 PgBouncer Strategy
Supabase provides PgBouncer in transaction pooling mode. This is critical for BullMQ workers and the NestJS API tier which otherwise would each hold a persistent connection per pod instance. Transaction mode releases the connection to the pool after each transaction commit, enabling far higher concurrency than the 100-connection default PostgreSQL limit would allow.
Implication: DO NOT use PostgreSQL advisory locks, LISTEN/NOTIFY, or session-scoped prepared statements with PgBouncer in transaction mode. Each transaction may be routed to a different server connection. Use application-level distributed locking via Redis SETNX for cross-request coordination.
6. Redis + BullMQ Architecture
6.1 Redis Responsibilities (Ephemeral Only)
Redis is treated as ephemeral cache and coordination infrastructure. No durable business data is stored solely in Redis. If the Redis instance is wiped, the system must recover without data loss. This constraint drives the following design decisions:
Scan count buffer (Redis INCR): If Redis is lost, the session-counter-sync job detects drift > 10 and triggers a full recount from PostgreSQL contact_encounters. Maximum data inconsistency: 5 seconds of scan count.
JWT jti blocklist: If Redis is lost, revoked tokens become temporarily valid until they expire naturally. Acceptable because token TTL is 1 hour. Recovery: re-add active revocations from audit_events on Redis restart.
Rate limiting counters: Reset on Redis loss. Acceptable—attackers cannot exploit a brief window to exceed limits at meaningful scale before Redis recovers.
BullMQ job state: BullMQ persists job data in Redis including job payloads, results, and retry state. While jobs in-flight during a Redis failure may be lost, the upstream systems (OCR submit, sync reconnect) are idempotent and will re-enqueue on the next request. Completed jobs are retained per COMPLETED_JOB_RETENTION policy (1000 jobs, 24hr TTL).

6.4 Idempotency Protection
Every BullMQ job carries a correlation_id and where applicable a client_idempotency_key. Worker implementation must be idempotent: re-processing the same job must produce the same outcome without side effects. Implementation pattern:
// Worker idempotency check (sync-ingest worker example)
async processJob(job: Job<SyncIngestPayload>) {
const key = job.data.client_idempotency_key;
// 1. Check Redis cache (O(1), TTL 24h)
const cached = await this.redis.get(`idem:${key}`);
if (cached) return JSON.parse(cached);  // Already processed
// 2. Check DB (authoritative, handles Redis flush scenario)
const existing = await this.syncRepo.findByKey(key);
if (existing) {
await this.redis.setex(`idem:${key}`, 86400, JSON.stringify(existing));
return existing;
}
// 3. Process and persist atomically
const result = await this.processCapture(job.data);
await this.redis.setex(`idem:${key}`, 86400, JSON.stringify(result));
return result;
}

7.2 JWT Strategy
Authentication uses RS256 asymmetric JWT issued by Supabase Auth. The access token (1-hour TTL) carries claims: sub (user UUID), org (organization UUID), role (employee | manager | super_admin), jti (unique token ID for blocklist).
Refresh token rotation: 30-day TTL, single-use, rotated on every /auth/refresh call. Previous refresh token invalidated immediately via jti blocklist in Redis. Refresh token stored encrypted in mobile secure storage (Android Keystore). On refresh, both new access token and new refresh token returned. Old access token added to jti blocklist with TTL equal to remaining access token lifetime.
Offline behavior: JWT persisted locally in mobile secure storage. App functions offline without re-auth until token expires. On expiry while offline, JWT remains locally persisted for read access to SQLite data; re-auth required before next API call on reconnect.

8.3 Offline-First Architecture
SQLite (via react-native-quick-sqlite) stores the pending_captures table locally. Schema mirrors the server sync_queue contract:
CREATE TABLE IF NOT EXISTS pending_captures (
id                      TEXT PRIMARY KEY,   -- local UUID
client_idempotency_key  TEXT NOT NULL UNIQUE,
capture_mode            TEXT NOT NULL,
session_id              TEXT,
image_local_uri         TEXT,               -- local file path
image_storage_path      TEXT,               -- set after S3 upload
ocr_data                TEXT NOT NULL,      -- JSON blob
qualifier               TEXT,
encounter_type          TEXT,
lead_note               TEXT,
follow_up_date          TEXT,
status                  TEXT DEFAULT 'pending',  -- pending|syncing|synced|failed|dead_letter
retry_count             INTEGER DEFAULT 0,
created_at              TEXT NOT NULL,      -- ISO 8601
synced_at               TEXT
);
Why SQLite: On-device relational store with ACID guarantees. Survives app restarts, device reboots. FIFO drain order maintained by created_at index. Supports complex conflict state tracking via status column. Alternative (AsyncStorage) lacks transactional semantics required for reliable FIFO drain.
8.4 OCR Integration
Google ML Kit Text Recognition V2 is bundled as a native module (react-native-mlkit-ocr or custom bridge). Key implementation constraints:
ML Kit model must be bundled with the APK (not downloaded at runtime) to support fully offline operation
Quality gate runs before ML Kit: Laplacian variance for blur detection, aspect ratio for tilt, pixel count for resolution
Auto-shutter: confidence polling at 200ms intervals; fires when overall confidence > 0.90 sustained for 800ms
Field extraction: name, company, title, emails[], phones[], website, address — each with confidence score
Confidence < 0.60 overall after 3 retries triggers manual entry fallback, pre-filling any extracted fields
9. Admin Dashboard Architecture
9.1 React Architecture
React 18 with TypeScript, Vite build, Tailwind CSS. State management via TanStack Query (server state) + Zustand (UI state, auth). No Redux—unnecessary complexity for a data-fetching-centric dashboard.
Data fetching: TanStack Query with 30-second stale time for analytics, 5-second stale time for session stats (near-real-time). Polling enabled for session detail page (active sessions only). Pagination via cursor-based approach using TanStack Query's infinite queries.
Table virtualization: TanStack Virtual for contact lists and audit logs exceeding 500 rows. This is non-negotiable for the contacts list which may contain tens of thousands of records per organization.
9.2 Role-Based UI Rendering
Role-irrelevant UI elements are hidden, not disabled (per UX specification). Implementation: a PermissionContext provider at app root reads role from the decoded JWT. A usePermission(action) hook returns boolean. Components conditionally render based on this hook, not CSS visibility—hidden elements must not be in the DOM for security.
Sidebar items (Platform, OCR Monitoring, Tenant Management, Relationship Match Panel) are only rendered for super_admin. Manager-only items (Analytics, Audit Log, Users) are only rendered for manager and super_admin. Employee users see only home and contact management.
9.3 Export Workflow
Export is an async operation. UI flow: (1) Manager configures export filters (session, qualifier, date range); (2) POST to API returns export_id immediately; (3) UI polls GET /exports/:id every 2 seconds until status = ready; (4) signed download URL (1-hour TTL) presented as download button. Rate limit enforced server-side (5 exports/hr per org) and displayed in UI as countdown.

11.3 Kubernetes vs ECS Recommendation
Recommendation: Kubernetes (EKS or GKE). Rationale: CardVault has 8 independent BullMQ worker deployments each requiring separate HPA (Horizontal Pod Autoscaler) based on queue depth metrics. Kubernetes provides the Keda (Kubernetes Event-Driven Autoscaler) operator which natively scales worker pods based on BullMQ queue depth, a capability ECS Task scaling does not cleanly support without custom CloudWatch metrics. The additional operational complexity of Kubernetes is justified by this requirement.
API tier: Deployment with HPA (CPU/memory, target 70% CPU)
Worker tier: 8 Deployments (one per queue), each with KEDA ScaledObject targeting queue depth
KEDA trigger: BullMQ queue length via Redis metrics; scale up at depth > 50, scale down at depth < 5
Dashboard: Static assets on Vercel or Cloudflare Pages (zero-ops, global CDN)

12.2 Partitioning Strategy
Table partitioning is planned but not required at launch. Partition implementation should be triggered by row count thresholds:
contacts: HASH partitioning on organization_id (8 partitions) — trigger at 100M rows total. Keeps per-tenant queries on a single partition, avoiding cross-partition scatter.
contact_encounters: RANGE partitioning on created_at (yearly) — trigger at 50M rows. Time-range queries for relationship history stay within a single partition.
audit_events: RANGE on created_at (monthly) from day one — legal retention requirement drives this. Old monthly partitions moved to read-only replica after 90 days.
ocr_jobs: RANGE on created_at (quarterly) — trigger at 20M rows. Enables dropping old partitions once jobs are past retention period.

18.2 Data Integrity Risks
Concurrent contact creation (same person scanned by two users simultaneously): Deduplication worker handles this post-creation. Brief window where two contact records exist is acceptable — dedup merges and audit logs the decision within seconds.
Scan counter drift > 10: Automated detection via session-counter-sync job. Full PostgreSQL recount triggered automatically. Engineering alert fired for investigation.
Audit log gaps: audit-writes queue has 5 retry attempts before Sentry alert and direct DB fallback. This is the last-resort path — the queue is designed to never lose audit events.
OCR data corruption (wrong field mapping): Confidence scoring and manual review flow catch low-confidence extractions. Field-level confidence < 0.50 requires mandatory correction before save. Original image always stored for reprocessing.

1. Executive Summary
CardVault is a multi-tenant, enterprise-grade SaaS platform engineered to digitize, centralize, and operationalize business card data across sales organizations. Built on a React Native mobile foundation with a Node.js/NestJS backend powered by Supabase's managed PostgreSQL infrastructure, CardVault eliminates the systemic inefficiencies associated with manual contact management, fragmented contact ownership, and data silos endemic to large sales teams.
The platform delivers a seamless capture-to-contact workflow: a sales employee photographs a physical business card, Google ML Kit OCR extracts structured contact data with confidence scoring, the backend deduplication engine resolves or merges against existing records, and the contact is immediately available organization-wide through a real-time searchable interface. Original card images are preserved in secure object storage with full lifecycle management.
CardVault v3.0 extends this foundation with context-aware capture. Every card scan now begins with context selection — Visitor (attending an event), Exhibitor (collecting leads at a booth), or Quick Capture (ad-hoc encounters). Each mode delivers a distinct capture experience, data model, and downstream workflow, transforming CardVault from a digitization tool into a relationship intelligence platform.
CardVault is designed for internal enterprise distribution on Android (APK-based deployment), with a centralized admin dashboard for tenant management, OCR quality monitoring, audit compliance, and storage governance. v3.0 is backward-compatible with all v1.0 data — all changes are additive.

2. Product Vision
CardVault's north-star vision is to be the definitive enterprise system of record for first-contact relationship data — and to extend that record to include the full context of every relationship's origin.
2.1 Vision Statement
"To eliminate the gap between the physical exchange of a business card and the organization's unified, actionable knowledge of that contact relationship — ensuring that every connection is stored with the context, qualification, and history needed to convert it into revenue."
2.2 The Core Insight
Every other contact capture application treats the business card as the unit of value. CardVault v3.0 treats the relationship context as the unit of value. A card captured at a trade show booth by an exhibitor is a sales lead. A card captured at a competitor's booth by a visitor is competitive intelligence. A card captured on a flight is a personal introduction. These are fundamentally different assets requiring different workflows.
2.3 Design Principles
Context First: Capture mode selection is the first action — before the camera opens.
Capture First: The mobile interface must enable frictionless card capture in under 30 seconds.
Zero Friction per Mode: Each mode is optimized for its specific environment — exhibitors need speed, visitors need organization, quick capture needs simplicity.
Data Fidelity: OCR extraction accuracy and original image preservation are non-negotiable.
Relationship Intelligence: The same person appearing in multiple events is not a duplicate problem — it is a relationship history to be surfaced and celebrated.
Lead-Native for Exhibitors: The exhibitor mode is a first-class lead capture tool, not an afterthought.
Organization-Wide Intelligence: No contact should be siloed to an individual user.
Enterprise Security by Default: All data is encrypted, access-controlled, and audited.
Offline Resilience: All three modes function fully offline; event sessions sync on reconnection.
Operational Observability: Administrators must have full visibility into system health and data quality.

6. Functional Requirements
6.1 Context Selection — Home Screen
This is the most important UX change in v3.0. The home screen is the gateway to all three capture modes. Design must make mode selection feel like a natural first step, not overhead.
FR-HOME-001: Home screen displays three primary capture mode cards prominently above the recent contacts feed.
FR-HOME-002: Mode cards display: icon, mode name, one-line description, and active session indicator (if a session is already open).
FR-HOME-003: User can have one active session per mode simultaneously.
FR-HOME-004: Active session indicator shows session name and scan count; tapping resumes the session directly.
FR-HOME-005: Recent contacts feed below mode cards shows last 10 contacts captured (across all modes), with mode badge indicator.
FR-HOME-006: Search bar on home screen searches org-wide across all modes and sessions.
FR-HOME-007: Mode selection is not mandatory — a 'Skip / Quick Scan' fallback allows direct OCR without mode context (backwards compatibility).
6.2 Visitor Mode
6.2.1 Event Session Management
FR-VIS-001: User taps 'Visitor' on home screen to enter Visitor mode.
FR-VIS-002: If no active Visitor session exists, user is presented with: 'Join existing event' (org-shared sessions) or 'Create new event'.
FR-VIS-003: Create event flow: event name (required), event type template (Trade Show / Conference / Product Demo / Networking / Other), date (defaults to today), location (optional, GPS-assisted).
FR-VIS-004: Event type templates pre-fill the session with relevant default tags and suggested note prompts.
FR-VIS-005: Org-shared sessions: sessions created by any org member are visible to all org members to join; joining adds your scans to it.
FR-VIS-006: User can switch between sessions mid-event without losing scan history.
FR-VIS-007: Session can be closed/ended by creator or manager; closed sessions are read-only but fully viewable.
6.2.2 Visitor Capture Flow
FR-VIS-010: Camera opens within session context; session name displayed in header during capture.
FR-VIS-011: After OCR review, user sees optional 'Note about this contact' field.
FR-VIS-012: Contact is saved and linked to the active session via event_session_id foreign key.
FR-VIS-013: Session view screen shows all contacts captured in that session as a scrollable list with thumbnails.
FR-VIS-014: Visitor sessions are not lead-qualified by default (no Hot/Warm/Cold); this is the Exhibitor mode differentiator.
6.3 Exhibitor Mode
Exhibitor mode is the highest commercial-value feature in CardVault v3.0. It positions the platform as a direct replacement for physical lead capture devices rented at trade shows (e.g., Swapcard, CompuLead, iLeads) which charge $200–800/event/device.
6.3.1 Booth Event Session Management
FR-EXH-001: User taps 'Exhibitor' on home screen to enter Exhibitor mode.
FR-EXH-002: Create booth event: booth name / event name (required), event type (Trade Show / Conference / Roadshow / Other), date range (multi-day support), team members assigned to booth.
FR-EXH-003: Multiple team members can be assigned to the same booth session; all scans aggregate under the single booth session.
FR-EXH-004: Booth session displays real-time scan counter visible to all assigned team members.
6.3.2 High-Volume Scan & Lead Qualification
FR-EXH-010: After OCR extraction and review, Exhibitor mode presents the Lead Qualifier screen before saving.
FR-EXH-011: Lead Qualifier UI: three large tap targets — HOT (red), WARM (amber), COLD (blue) — optimized for one-thumb operation.
FR-EXH-012: Optional quick note field below qualifier (max 160 characters); voice-to-text supported.
FR-EXH-013: Lead qualifier rating and note are stored in lead_qualifier (enum) and lead_note (text) fields on the contact record.
FR-EXH-014: After save, camera immediately reopens for next scan (high-volume mode — no return to home screen between scans).
FR-EXH-015: Session summary screen shows live: total scans, HOT count, WARM count, COLD count, unqualified count.
FR-EXH-016: Booth session lead list exportable to CSV segmented by qualifier tier; export available to Manager role+.
FR-EXH-017: Lead qualifier can be edited post-scan from contact detail screen.
6.4 Quick Capture Mode
6.4.1 Encounter Type Tagging
FR-QC-001: User taps 'Quick Capture' on home screen; no session creation required.
FR-QC-002: Encounter type selector presented before camera opens: Flight, B2B Meeting, Airport / Transit, Dinner / Social, Referral, Conference Hallway, Other.
FR-QC-003: Encounter type tag is stored in encounter_type field on contact record.
FR-QC-004: 'Other' type allows free-text label (max 40 chars).
FR-QC-005: Encounter type selection defaults to last-used type for repeat users.
6.4.2 Quick Capture Flow
FR-QC-010: After OCR review, user sees: encounter type (pre-filled, editable), quick note field (max 200 chars), optional follow-up date selector.
FR-QC-011: Follow-up date triggers a local notification on the selected date.
FR-QC-012: Quick Capture contacts are stored in the org-wide contact database with encounter_type and no event_session_id (null session).
FR-QC-013: Quick Capture contacts are visible in org-wide search; filterable by encounter_type.
FR-QC-014: Quick Capture mode has no team assignment — all captures are attributed to the individual user.
6.5 Cross-Event Relationship History
This feature transforms the duplicate detection problem into a relationship intelligence feature. Instead of asking 'is this a duplicate?', the system asks 'when did you last interact with this person, and in what context?'
FR-REL-001: When a new scan matches an existing contact (by email, phone, or name+company fuzzy match), the system does NOT immediately prompt for merge.
FR-REL-002: Instead, the system surfaces a Relationship History card: 'You've met [Name] before — at [Session Name] on [Date], captured by [User Name]'.
FR-REL-003: User is presented with three options: (a) 'Add new encounter to existing contact'; (b) 'Create separate contact'; (c) 'View existing contact'.
FR-REL-004: Option (a) 'Add new encounter' appends a new entry to the contact_encounters table with new session, date, qualifier, and note.
FR-REL-005: Contact detail screen shows full encounter history timeline: date, session name, captured by, mode, qualifier, note.
FR-REL-006: Encounter history is visible to all org members with access to the contact.
FR-REL-007: Cross-org relationship history (same person captured by two different org members in different sessions) is surfaced: 'Also met by [Colleague Name] at [Event] on [Date]'.
6.6 Authentication & Authorization
CardVault's authentication layer is built on Supabase Auth, leveraging JWT-based session management with tenant-scoped token claims.
FR-AUTH-001: Users authenticate via email/password with bcrypt hashing (min cost factor 12).
FR-AUTH-002: JWT access tokens expire after 1 hour; refresh tokens expire after 30 days with rotation on use.
FR-AUTH-003: All JWTs carry organization_id and role claims; backend middleware validates both on every protected route.
FR-AUTH-004: Session invalidation propagates within 60 seconds via refresh token revocation.
FR-AUTH-005: Supabase Row Level Security (RLS) policies enforce organization_id isolation at the database layer.
FR-AUTH-006: Failed authentication attempts are rate-limited to 5 attempts per 15-minute window per IP.
FR-AUTH-007: All authentication events are written to the audit_events table.
FR-AUTH-008: SSO integration via SAML 2.0 / OIDC is a Phase 2 requirement (post-MVP).
FR-AUTH-009: Event session creation requires Employee role minimum; session deletion requires Manager role.
FR-AUTH-010: Booth session team assignment is managed by the session creator or Manager role.
6.7 OCR Processing System
The OCR pipeline is the core value-creation engine of CardVault. It must be resilient, accurate, and observable.
FR-OCR-001: Camera integration via React Native Vision Camera with real-time frame preview, auto-focus, and flash control.
FR-OCR-002: Pre-capture image quality gate: minimum 800x600px resolution; blur detection via Laplacian variance; tilt angle detection.
FR-OCR-003: Google ML Kit Text Recognition V2 processes captured image on-device for initial field extraction.
FR-OCR-004: OCR output is parsed by a field classifier that maps extracted text blocks to: full_name, title, company, phone (multiple), email (multiple), address, website, linkedin.
FR-OCR-005: Each extracted field receives a confidence_score (0.0–1.0). Fields below 0.70 threshold are flagged for manual review.
FR-OCR-006: If overall extraction confidence (mean of field scores) is below 0.60, the user is prompted to retake or manually enter.
FR-OCR-007: Up to 3 automatic OCR retries are performed on extraction failure before entering manual correction flow.
FR-OCR-008: All OCR attempts, confidence scores, and outcomes are persisted to the ocr_jobs table for analytics.
FR-OCR-009: Manual correction UI presents extracted fields in an editable form with original card image displayed for reference.
FR-OCR-010: Batch OCR reprocessing is available from the Super Admin dashboard for failed or low-confidence records.
FR-OCR-011: OCR job record now includes session_id and capture_mode fields for analytics attribution.
6.8 Business Card Image Storage
FR-IMG-001: Original full-resolution JPEG image is uploaded to Supabase Storage under org_id/sessions/session_id/contact_id/original.jpg (or org_id/quick/contact_id/ for Quick Capture).
FR-IMG-002: A compressed thumbnail (max 400x300px, JPEG quality 70) is generated server-side and stored at .../thumbnail.jpg.
FR-IMG-003: Images are encrypted at rest using AES-256; Supabase Storage enforces access via signed URLs with 1-hour expiry.
FR-IMG-004: Image metadata (file_size, mime_type, dimensions, storage_path, upload_timestamp, uploader_id) is stored in the card_images table.
FR-IMG-005: Image retrieval requires valid JWT with matching organization_id claim; cross-tenant image access is prohibited.
FR-IMG-006: Image retention policy is configurable per organization (default: indefinite; configurable to 12, 24, 36 months).
FR-IMG-007: Deleted contacts result in image soft-delete with 30-day recovery window; hard deletion purges storage objects.
FR-IMG-008: Original images support reprocessing: admin can trigger re-OCR from stored original without re-capture.
6.9 Contact Management
FR-CON-001: Full CRUD operations on contact records via REST API with field-level validation.
FR-CON-002: Contact fields: full_name, first_name, last_name, title, company, department, phones[], emails[], address, website, linkedin_url, notes, tags[], captured_by, organization_id, is_verified, capture_mode, event_session_id, encounter_type, lead_qualifier, lead_note, created_at, updated_at.
FR-CON-003: Duplicate detection runs at point of ingestion using normalized email, phone, and (name + company) fuzzy match with Levenshtein distance threshold of 0.85.
FR-CON-004: Duplicate candidates trigger the Relationship History flow (FR-REL-001 through FR-REL-007) rather than an immediate merge prompt.
FR-CON-005: Contact merge preserves all source field values as a versioned history in the contact_merge_history table.
FR-CON-006: Full-text search across name, company, email, phone, tags using PostgreSQL tsvector with GIN indexing.
FR-CON-007: Contacts are visible organization-wide to all authenticated members of the same tenant.
FR-CON-008: Contact history log tracks all field changes with timestamp and actor_id.
FR-CON-009: Tags support free-form labels and org-defined taxonomy; tags are searchable and filterable.
FR-CON-010: Bulk export to CSV/JSON for contacts matching a filter set; export events are audit-logged.
FR-CON-011: Contact list views support filtering by: capture_mode, event_session_id, lead_qualifier, encounter_type, date range.
FR-CON-012: Contact detail screen shows encounter history timeline (FR-REL-005).
FR-CON-013: Bulk lead qualifier update from session view (e.g., select 10 contacts, mark all WARM).
6.10 Multi-Tenant SaaS Architecture
FR-MT-001: All database tables include organization_id (UUID) as a non-nullable foreign key.
FR-MT-002: Supabase RLS policies enforce: SELECT/INSERT/UPDATE/DELETE restricted to rows where organization_id = auth.jwt()->>'organization_id'.
FR-MT-003: Tenant provisioning creates: organization record, default admin user, storage bucket policy, RLS enforcement validation.
FR-MT-004: Cross-tenant API requests are rejected at the middleware layer with 403 Forbidden before query execution.
FR-MT-005: Storage bucket paths are namespaced by organization_id; bucket-level policies prevent cross-tenant object access.
FR-MT-006: Tenant deprovisioning flow: contact data export, 30-day grace period, hard deletion of all tenant data and storage objects.
FR-MT-007: event_sessions table includes organization_id; RLS policies enforce tenant isolation on all session data.
6.11 Admin Dashboard
FR-ADMIN-001: Web-based dashboard built with React + TypeScript, served independently from the mobile application.
FR-ADMIN-002: Super Admin view: all-tenant KPIs, per-tenant drill-down, system health, OCR pipeline monitoring.
FR-ADMIN-003: Organization Manager view: org-scoped analytics, user management, export management, duplicate queue.
FR-ADMIN-004: Audit log browser with full-text search, time range filter, actor filter, and event type filter.
FR-ADMIN-005: OCR monitoring panel: daily scan volumes, average confidence scores, failure rates, reprocessing queue.
FR-ADMIN-006: Storage monitoring: per-tenant usage, total platform usage, growth trends, cost estimation.
FR-ADMIN-007: Duplicate management queue: pending merge decisions, bulk auto-merge for high-confidence duplicates.
FR-ADMIN-008: User management: invite, deactivate, role change, session invalidation.
FR-ADMIN-009: Event Sessions panel: list of all org sessions, mode type badge, scan count, date range, creator, status (active/closed), export button.
FR-ADMIN-010: Lead Funnel panel (Exhibitor sessions only): HOT/WARM/COLD breakdown by session, trend over time.
FR-ADMIN-011: Encounter Type panel (Quick Capture): breakdown of encounter types across org, time trend.
FR-ADMIN-012: Relationship History audit: list of cross-event relationship matches surfaced, user decisions (add encounter / create new / view existing).
6.12 Offline Support
FR-OFF-001: Business card capture, image storage (local), and OCR processing (on-device) function fully offline — all three modes.
FR-OFF-002: Offline captures are queued in a local SQLite database with sync_status: pending; includes session_id, capture_mode, lead_qualifier, encounter_type, follow_up_date.
FR-OFF-003: Background sync triggers on network restoration, WiFi connection, and manual user trigger.
FR-OFF-004: Sync conflict resolution policy: server-wins for concurrent edits; offline creates are always accepted.
FR-OFF-005: Sync queue UI displays pending, in-progress, and failed items with retry affordance.
FR-OFF-006: Maximum offline queue size: 500 pending captures (configurable per org); user is warned at 80% capacity.
FR-OFF-007: Active event sessions are cached locally; new captures under a session are queued in the local SQLite pending_captures table with session_id reference.
FR-OFF-008: Session scan counter in offline mode is tracked locally and reconciled on sync.
FR-OFF-009: Encounter type and lead qualifier values are stored locally; no network required for qualification flow.

10.4 Rate Limiting Strategy
Global rate limit: 1,000 requests per 15 minutes per authenticated user.
OCR submission endpoint: 30 requests per minute per user.
Image upload endpoint: 20 requests per minute per user.
Export endpoint: 5 requests per hour per user.
Rate limit state stored in Redis (Upstash) for distributed enforcement across horizontal scale.
429 responses include Retry-After header.
10.5 API Versioning
API versioning follows URI path versioning: /api/v1/. Deprecation policy: minimum 12-month support window for retired versions with documented migration path. Breaking changes require a major version increment.
10.6 Error Handling Architecture
All exceptions are caught by GlobalExceptionFilter and normalized to a standard error response envelope with status, code, message, correlationId, and timestamp fields. HTTP status codes follow RFC 7231. Custom application error codes (e.g., OCR_CONFIDENCE_TOO_LOW, DUPLICATE_CONTACT_DETECTED, SESSION_CLOSED) are surfaced in the code field for client-side handling.
11. Database Design
11.1 Schema Overview
CardVault's PostgreSQL schema is organized around six functional domains: Identity & Tenancy, Contact Management, Event Sessions & Encounters, OCR Processing, Image Storage, and Audit & Compliance. All tables use UUID v4 primary keys, created_at/updated_at timestamps, organization_id foreign keys, and soft deletes via deleted_at.
v3.0 adds three new tables (event_sessions, session_members, contact_encounters) and four new columns to the contacts table. All v1.0 tables are unchanged in structure — changes are fully additive and backward-compatible.

12.2 Image Upload Lifecycle
Mobile app captures image; pre-validation checks resolution (min 800x600) and blur (Laplacian variance > 100).
Client compresses JPEG to quality 85 using react-native-image-manipulator before upload.
Client requests a signed upload URL from the backend: POST /api/v1/images/upload-url.
Backend validates JWT, creates card_images record with status=pending, generates Supabase Storage signed upload URL (5-minute TTL).
Client uploads directly to Supabase Storage via signed URL (bypasses backend for bandwidth efficiency).
On upload completion, backend image processor queue job: validates file integrity, generates thumbnail, extracts EXIF metadata, updates card_images record to status=complete.
OCR job is enqueued for processing against the confirmed image.

14. OCR Processing Workflow
14.1 End-to-End OCR Pipeline
PRE-CAPTURE VALIDATION: Real-time blur detection (Laplacian variance threshold > 100); minimum frame resolution 800x600px; card boundary detection via ML Kit Object Detection API.
IMAGE CAPTURE & PREPROCESSING: Capture full-resolution JPEG; apply contrast normalization, deskewing, noise reduction; compress to JPEG quality 85.
ON-DEVICE OCR EXECUTION: Google ML Kit Text Recognition V2 processes image; returns TextBlocks with bounding boxes and confidence scores.
FIELD CLASSIFICATION: Regex + heuristic rules classify text blocks to field types. Email: RFC 5322 compliant regex. Phone: E.164 normalization with libphonenumber-js. Name heuristic: proper-case two-word patterns in header position.
CONFIDENCE EVALUATION: Per-field confidence computed from ML Kit confidence + classification certainty. Mean confidence < 0.60: trigger retry (up to 3 attempts). Per-field confidence < 0.70: flag field for manual review.
MODE-SPECIFIC POST-OCR SCREEN: Visitor mode: standard review form + optional note. Exhibitor mode: review form + mandatory Lead Qualifier (Hot/Warm/Cold) + optional 160-char note. Quick Capture mode: review form + encounter type + optional 200-char note + optional follow-up date.
BACKEND SUBMISSION — POST /api/v1/ocr/submit: Extended payload includes capture_mode, session_id, lead_qualifier, lead_note, encounter_type, encounter_label, follow_up_date.
RELATIONSHIP HISTORY CHECK: relationship-matching queue job runs immediately post-ingest. If match found: push notification to mobile client with relationship history card. User chooses: Add encounter / Create new / View existing.
CONTACT RECORD CREATION / ENCOUNTER APPEND: Contact and contact_encounters records written; session scan_count incremented; analytics events fired.

16. Security Architecture
16.1 Defense-in-Depth Model
Transport Security
TLS 1.3 enforced on all client-server communications; TLS 1.2 minimum fallback.
HSTS headers enforced with 1-year max-age.
Certificate pinning implemented in React Native application for API domain.
All Supabase Storage signed URLs served over HTTPS only.
Authentication Security
RS256 asymmetric JWT signing; private key stored in environment secrets manager.
JWT claims include: sub (user_id), org (organization_id), role, iat, exp, jti.
Token revocation via jti blocklist stored in Redis with TTL matching token expiry.
Refresh token rotation: every use issues a new refresh token; previous is immediately invalidated.
Data Security
PostgreSQL data encrypted at rest (AES-256) via Supabase managed encryption.
Business card images encrypted at rest in Supabase Storage.
PII fields (emails, phones) are not indexed in plaintext in any secondary search index.
Lead qualifier ratings (Hot/Warm/Cold) are treated as commercially sensitive data; export endpoints require Manager role and generate a full audit event.
Application Security
All API inputs validated via class-validator DTOs before reaching service layer.
SQL injection prevention: Supabase JS client uses parameterized queries exclusively.
XSS prevention: admin dashboard uses React with no dangerouslySetInnerHTML usage.
CSRF protection via SameSite=Strict cookies and Origin header validation.
Dependency vulnerability scanning via npm audit in CI pipeline; CRITICAL severity blocks deployment.
v3.0 Security Additions
Session Isolation: Event sessions are tenant-scoped via organization_id; RLS policies enforce isolation at the database layer.
Session Member Access: Only users in the session_members table (or Manager/Super Admin) can add contacts to a closed session or view Exhibitor lead exports.
Encounter Note Privacy: Lead notes and encounter notes stored as plain text in PostgreSQL; subject to the same AES-256 encryption at rest as all PII fields.
Follow-up Date Notifications: Generated as local device notifications only; no notification content transmitted to third-party push services without explicit opt-in.
Relationship History Access: Cross-org relationship history visible only within the same tenant; contact data from other tenants is never surfaced.
Compliance Controls
Immutable audit trail for all data access and modification events.
Data retention policies configurable per organization for GDPR Article 17 compliance.
Right-to-erasure workflow: contact deletion triggers image deletion and audit log flagging.
Export access logged with actor identity, timestamp, filter criteria, qualifier filter, and record count.

19.3 Session Counter Reconciliation
Local scan_count is incremented immediately on every offline capture.
On sync, the session-counter-sync queue job reconciles local count against server count.
Conflict policy: server count + number of offline captures not yet synced = reconciled total.
Duplicate submission prevention: each capture carries a client_idempotency_key (UUID generated at capture time); server rejects duplicate submissions with 200 (already processed).

24. Scalability Considerations
24.1 Database Scalability Path
Horizontal read scaling: Supabase read replicas for analytics queries and admin dashboard.
Connection pooling: PgBouncer (transaction mode) to support high concurrent API request counts without exhausting PostgreSQL connection limits.
Partitioning strategy at 100M+ contacts: partition contacts table by organization_id hash or range on created_at.
Contact Encounters at Scale: The contact_encounters table grows faster than contacts (multiple encounters per contact). At 10M contacts with average 2 encounters each: 20M rows. Partitioning by created_at year recommended at 50M+ rows.
JSONB field promotion: frequently queried JSONB paths promoted to generated columns with indexes.
24.2 Application Scalability
Stateless NestJS API instances: horizontally scalable behind a load balancer; no in-process session state.
Redis-backed rate limiting and session invalidation ensures consistency across all API instances.
BullMQ workers scaled independently from API tier based on queue depth metrics.
Session Scan Counter Hot Path: Exhibitor booths can generate 1+ scan/second across team members. scan_count increments are buffered in Redis and flushed to PostgreSQL in batches every 5 seconds to avoid high-frequency row-level lock contention.
Relationship Matching Performance: fuzzy string matching scoped to organization_id; trgm index on contacts.search_vector pre-built. At 10M contacts per tenant, query is bounded by org scope.
24.3 Storage Scalability
Supabase Storage (S3-compatible) scales to unlimited object count; storage costs are tiered.
CDN-backed delivery ensures image retrieval latency is independent of origin storage load.
Aggressive compression policy targets < 150KB per original image after JPEG quality 85 compression.
Event Sessions at Scale: At 1M+ sessions per large enterprise tenant, session queries are protected by composite index. Sessions older than 2 years are candidates for archival.
At 10M contacts with one image each: estimated 1.5TB storage; $30–45/month at S3 rates with compression.
